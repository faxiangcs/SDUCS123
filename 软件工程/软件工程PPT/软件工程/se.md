## 第一章

### 1.软件危机

1. **成本和进度估计不准确**：软件开发成本和进度的估计常常出现较大偏差，导致项目超支和延期。
2. **软件质量不可靠**：软件产品存在质量问题，可靠性低，容易出现故障和错误。
3. **用户不满意**：用户对已完成的软件系统经常表示不满，因为软件往往不能完全满足他们的需求。
4. **软件维护困难**：软件产品难以维护，对新环境的适应性差，难以添加新功能或修复存在的问题。
5. **缺乏适当的文档资料**：软件开发过程中缺乏必要的文档支持，使得维护和后续开发变得困难。
6. **软件成本占比上升**：软件成本在计算机系统总成本中所占的比例逐年上升，对企业和组织构成经济负担。
7. **生产率提高缓慢**：软件开发的生产率提高速度跟不上计算机应用的普及和深入。

### 2.软件工程

**软件工程**是**以工程化的方法和管理手段来开发和维护高质量软件**的学科，旨在提高软件的**可靠性、可维护性和开发效率**，并解决软件危机问题。

### 3.软件生存周期

**软件生存周期指的是**软件从需求提出到最终退役的整个过程**，通常包括以下 **6 个阶段**：

1. **可行性分析**
2. **需求分析**：收集、分析用户需求，形成需求文档。
3. **系统设计**：制定软件架构、模块划分、数据流设计等。
4. **实现（编码）**：根据设计文档进行编程。
5. **集成与测试**：单元测试、集成测试、系统测试，保证软件质量。
6. **运行与维护**：软件上线后进行维护、优化和升级

### 4.常用软件过程模型

#### **1. 瀑布模型**

**特点**：线性顺序开发，每个阶段完成后才能进入下一阶段。
**适用场景**：需求稳定、变更少的项目，如政府、军事系统。
**优点**： 结构清晰，易管理
**缺点**： 需求变更困难，后期修改成本高

**流程**：需求分析 → 设计 → 编码 → 测试 → 运行与维护

------

#### **2. 快速原型模型**

**特点**：先开发一个**原型（Prototype）**，让用户体验并反馈，再完善正式系统。
**适用场景**：需求不明确、用户参与度高的项目，如新产品开发。
**优点**：
✔ 用户早期参与，提高满意度
✔ 适应需求变化
**缺点**：
✘ 可能导致开发周期延长
✘ 原型可能无法直接复用

📌 **流程**：需求分析 → 设计原型 → 用户反馈 → 迭代优化 → 开发最终版本

------

#### **3. 增量模型**

**特点**：把软件拆分为多个小部分，每次**增量开发新功能**。
**适用场景**：需求可能变更，需要分阶段交付的项目，如 Web 应用。
**优点**：
✔ 交付快，可用性高
✔ 降低风险，灵活性强
**缺点**：
✘ 总体成本可能较高
✘ 早期版本可能功能不完整

📌 **流程**：基础版本 → 增量 1 → 增量 2 → … → 最终版本

------

#### **4. 螺旋模型**

**特点**：结合**瀑布模型**和**增量模型**，强调**风险管理**。
**适用场景**：高风险、大型复杂系统，如航空航天、银行系统。
**优点**：
✔ 适用于复杂项目，风险可控
✔ 允许需求调整，灵活性高
**缺点**：
✘ 过程复杂，成本高
✘ 适合长期项目，不适合小型项目

📌 **流程**：需求分析 → 风险评估 → 设计 → 迭代开发 → 评估 → 下一轮迭代

------

#### **5. 喷泉模型**

**特点**：适用于**面向对象开发（OOP）**，强调**并行开发**，多个阶段可以**交叠进行**。
**适用场景**：面向对象的软件开发，如 Java/C++ 项目。
**优点**：
✔ 允许多个开发阶段并行，提高效率
✔ 适应面向对象编程，代码可复用
**缺点**：
✘ 需要良好的团队协作和工具支持

📌 **流程**：分析 & 设计 ⟶ 代码 & 测试 ⟶ 维护（多个阶段并行进行）

------

#### **6. Rational 统一过程**

**特点**：由 IBM Rational 公司提出，基于**迭代开发**，强调**架构设计、组件化、风险控制**。
**适用场景**：企业级软件开发，如 ERP、银行系统。
**优点**：
✔ 迭代开发，适应需求变化
✔ 强调架构设计，提高可维护性
**缺点**：
✘ 过程较复杂，适合大型团队

📌 **四个阶段**：

1. **初始阶段**（Inception）：确定需求，定义项目目标
2. **细化阶段**（Elaboration）：详细设计，定义架构
3. **构造阶段**（Construction）：编码和测试
4. **移交阶段**（Transition）：部署和维护

------

#### 7. 敏捷开发模型

**特点**：以**快速迭代**、**持续交付**和**客户协作**为核心，代表方法有 **Scrum、XP**。
**适用场景**：需求变动频繁、需要快速交付的项目，如互联网应用。
**优点**：
✔ 适应需求变化，交付速度快
✔ 用户参与度高，提高满意度
**缺点**：
✘ 需要高效团队协作，否则混乱
✘ 不适合大型、严格监管的项目

📌 **流程（Scrum 示例）**：产品需求 → 迭代开发（1~4 周）→ 评审 → 发布 → 下一个迭代
